# Claude Only Lite Windows 打包设计

## 目标

产出一个验证版本：在 macOS 上准备分发包，在 Windows 上仅依赖现有的 Node 24，通过双击 `vbs` 启动文件即可运行。

这个验证版本需要尽量保留 Claude Code 的核心使用流程，同时移除会阻碍跨平台分发的原生依赖和非关键功能。

## 约束

- Windows 目标运行环境仅有 Node 24
- 不要求本地数据库安装
- 不要求打包为 `.exe`
- 启动入口必须是 `vbs` 文件
- 第一阶段以“尽快验证可运行”为优先，不追求完整功能覆盖

## 范围

### 保留

- 仅保留 Claude Code Provider
- Claude 项目发现
- Claude 会话列表与聊天
- 文件树浏览
- 代码查看与编辑
- Git 面板
- 基础界面框架与 PWA 安装能力

### 移除

- Codex Provider
- Gemini Provider
- Cursor Provider 及 Cursor 会话导入能力
- 内置 Shell / PTY 终端
- 本地登录、密码认证、API Key 认证、用户数据库
- 多用户能力与依赖数据库的偏好设置流程

## 推荐方案

保留当前单体结构，也就是 `Express + Vite` 的前后端一体应用，但将其收敛为一个仅支持 Claude、无认证、无原生运行时依赖的精简版本。

这是当前改动最小、最有机会快速做出可运行验证包的方案，同时能保住这个项目的核心价值。

## 架构调整

### 后端

- 只保留 `claude` 相关 Provider 接线
- 删除仅服务于 `codex`、`gemini`、`cursor` 的路由与辅助逻辑
- 将认证中间件改为本地单机免登录模式
- 移除数据库初始化以及依赖数据库的认证/配置逻辑
- 移除基于 `node-pty` 的 Shell WebSocket 与会话管理

### 前端

- 移除 Provider 选择，默认固定为 Claude
- 隐藏或删除 Shell 标签页
- 隐藏或删除登录页与登录保护逻辑
- 移除仅适用于已删除 Provider 或认证流程的设置项
- 保留项目、聊天、文件、Git 等主视图

### 打包

- Windows 上使用 `node server/index.js` 启动
- 提供 `start.cmd` 作为命令行入口
- 提供 `start.vbs` 作为静默双击入口
- 启动后自动打开默认浏览器并访问 `http://127.0.0.1:3001`

## 依赖处理策略

需要从运行时中移除的依赖：

- `node-pty`
- `better-sqlite3`
- `sqlite3`
- `bcrypt`

`sharp` 不是当前验证版运行时必需依赖，也可以从分发链路中移除。

## 异常与兼容处理

- 如果 Claude CLI 或 Claude 项目目录不存在，尽量复用现有空状态界面
- 已删除路由不再由前端暴露入口
- 如果浏览器本地存储里残留旧的 Provider 状态，应用应安全回退到 Claude 默认值

## 测试策略

第一阶段以手工验证为主：

1. 在不依赖原生运行时模块的前提下完成依赖安装
2. 在 macOS 上启动服务
3. 确认应用加载后默认进入 Claude Only 模式
4. 确认聊天、文件树、编辑器、Git 面板可以正常渲染
5. 确认不再暴露 Shell UI
6. 确认不再要求登录
7. 确认 Windows 启动脚本可在 Node 24 环境下拉起服务

## 风险

- 已删除的 Provider 逻辑可能仍被共享状态或共享组件间接引用
- 去掉认证后，启动流程、WebSocket 连接与路由保护可能会出现连锁影响
- Git 与文件编辑能力可能仍间接依赖原本由认证或数据库初始化提供的状态

## 本次验证版的非目标

- 不支持 Codex、Gemini、Cursor
- 不支持内置终端
- 不保证完整保留原项目所有设置项
- 第一阶段不处理数据库迁移兼容问题
