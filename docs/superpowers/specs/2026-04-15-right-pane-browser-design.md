# 右侧统一内容面板与内嵌浏览器设计

**日期**: 2026-04-15  
**作者**: Codex  
**状态**: 已确认

## 概述

当前主界面右侧区域主要承载代码编辑器，Markdown 预览能力也依附在编辑器内部。用户希望把这块区域升级为统一的内容面板，新增一种“浏览器”类型，用来预览并交互网页。用户可以在右侧直接输入地址打开网页，也可以从文件区或对话区点击 `html` 文件、网页链接后，统一在右侧浏览器区域打开。

本方案采用“统一右侧面板”路径：将当前编辑器侧栏抽象为通用容器，在其内部支持 `code`、`markdown`、`browser` 三种视图类型。左侧聊天区、文件区、Git 区只负责发起打开动作，右侧面板统一承接展示与交互。

## 目标

1. 将右侧现有编辑器侧栏升级为统一内容面板。
2. 新增 `browser` 视图，支持本地开发页和普通外网页面。
3. 用户可在右侧地址栏输入 URL 并直接访问网页。
4. 点击聊天区、文件区中的网页链接时，默认在右侧浏览器打开，并自动切换到浏览器视图。
5. 点击项目中的 `html` 文件时，优先在右侧浏览器预览，而不是直接进入源码编辑。
6. 浏览器视图支持基础页面交互，包括点击、输入、滚动、表单操作。

## 非目标

1. 第一版不实现多标签页浏览器。
2. 第一版不实现浏览器扩展、下载管理器、开发者工具嵌入。
3. 第一版不实现复杂的跨页面共享会话策略。
4. 第一版不重做主界面整体左右布局，仅重构右侧面板的职责与状态模型。

## 用户流程

1. 用户在聊天区点击一个 `http://` 或 `https://` 链接。
2. 系统拦截默认新开页行为，调用统一的浏览器打开动作。
3. 右侧面板切换到 `browser` 视图并加载目标页面。
4. 用户可在右侧地址栏继续输入新地址、回车访问、后退、前进、刷新。
5. 用户在文件区点击一个普通代码文件时，右侧面板切换到 `code` 视图。
6. 用户在文件区点击一个 Markdown 文件时，右侧面板切换到 `markdown` 视图。
7. 用户在文件区点击一个 `html` 文件时，系统优先解析为可预览页面，并切换到 `browser` 视图。
8. 若 `html` 文件无法解析为可访问页面，则回退到 `code` 视图查看源码。

## 方案对比

### 方案 1：在现有编辑器侧栏中塞入浏览器模式

优点：

1. 改动面较小，接入速度快。

缺点：

1. 当前右侧区域的状态、文案、按钮语义都偏向代码编辑器。
2. 后续继续接入更多预览类型时会越来越耦合。
3. 无法自然统一 `html` 预览、Markdown 预览和代码编辑三种内容。

### 方案 2：把右侧重构为统一内容面板

优点：

1. 结构最符合“右侧统一打开”的产品目标。
2. 右侧状态模型更清晰，后续可扩展图片、PDF、设计稿等预览类型。
3. 可把现有空的 `preview` 容器和编辑器内的 Markdown 预览语义统一起来。

缺点：

1. 需要把当前 `editingFile` 升级为更通用的右侧面板状态。
2. 需要调整文件区、聊天区、Git 区的打开入口。

### 方案 3：新增一个独立浏览器侧栏，与编辑器并存

优点：

1. 对现有编辑器改动最少。

缺点：

1. 用户会面对两套右侧打开区域，心智不统一。
2. 文件和链接的打开规则容易分叉。
3. 不符合“统一在右边浏览器区域打开”的目标。

## 采用方案

采用方案 2：把右侧升级为统一内容面板，在一个容器中承载 `code`、`markdown`、`browser` 三种视图。

核心原则：

1. 左侧各业务区只负责触发“打开什么”，不关心“右侧如何展示”。
2. 右侧面板统一维护内容类型、来源、展示状态和控制行为。
3. `html` 文件和网页链接默认优先走浏览器视图。
4. 浏览器视图首版聚焦单页、可交互、可导航，不追求完整浏览器能力。

## 架构设计

### 状态模型

当前 `useEditorSidebar` 只维护 `editingFile`、宽度和展开状态。重构后建议升级为类似 `useRightPane` 的通用状态模型。

建议状态结构：

```ts
type RightPaneView = 'code' | 'markdown' | 'browser';

type RightPaneCodeTarget = {
  type: 'code';
  filePath: string;
  fileName: string;
  projectName?: string;
  diffInfo?: CodeEditorDiffInfo | null;
};

type RightPaneMarkdownTarget = {
  type: 'markdown';
  filePath: string;
  fileName: string;
  projectName?: string;
};

type RightPaneBrowserTarget = {
  type: 'browser';
  url: string;
  title?: string;
  source: 'address-bar' | 'chat-link' | 'file-html' | 'external-link';
  filePath?: string;
};

type RightPaneTarget =
  | RightPaneCodeTarget
  | RightPaneMarkdownTarget
  | RightPaneBrowserTarget;
```

保留现有宽度、展开、拖拽调节等侧栏能力，但把“当前打开目标”从单一文件提升为通用内容目标。

### 组件边界

建议拆分为以下结构：

1. `useRightPane`：统一管理右侧面板状态与动作。
2. `RightPane`：统一容器，负责标题栏、尺寸、关闭、展开、视图切换。
3. `RightPaneContentRouter`：根据 `target.type` 路由到具体视图。
4. `CodeEditorPane`：沿用现有代码编辑器。
5. `MarkdownPane`：承接当前 Markdown 预览相关展示。
6. `BrowserPane`：新增浏览器视图，负责地址栏、导航控制、页面容器。

### 入口统一

以下入口都应统一调用右侧面板动作，而不是各自直连不同 UI：

1. 文件区点击文件
2. Git 区点击文件
3. 聊天区工具结果中的文件打开动作
4. 聊天区 Markdown 链接点击
5. 右侧地址栏主动导航

建议暴露统一动作：

```ts
openFileInRightPane(filePath, diffInfo?)
openUrlInRightPane(url, source?)
closeRightPane()
toggleRightPaneExpand()
```

## 浏览器视图设计

### 第一版能力范围

浏览器视图应支持：

1. 地址栏输入 URL。
2. 回车访问。
3. 后退、前进、刷新、停止加载。
4. 页面加载中状态展示。
5. 点击、输入、滚动、表单交互。
6. 本地开发页与普通外网页面加载。

### 第一版不做

1. 多标签页。
2. 收藏、历史、下载中心。
3. 完整开发者工具嵌入。
4. 浏览器插件或脚本注入能力。

### 页面容器建议

渲染层建议优先复用桌面壳能力，例如 Electron `webview` 或项目现有宿主暴露的嵌入浏览器能力；若当前运行环境限制无法直接使用独立浏览器内核，再退而使用受限 `iframe` 方案。

选择原则：

1. 若要支持更稳定的外网页面交互、导航状态和站点兼容性，应优先使用宿主级嵌入浏览器能力。
2. 若只能使用 `iframe`，需接受部分站点因 `X-Frame-Options`、`CSP frame-ancestors` 而无法嵌入。
3. 无法嵌入时，界面应明确提示“该页面不允许内嵌预览”，而不是静默空白。

## 打开规则设计

### 普通代码文件

点击普通代码文件时，右侧面板切换到 `code` 视图，沿用现有编辑器行为。

### Markdown 文件

点击 Markdown 文件时，右侧面板切换到 `markdown` 视图。Markdown 视图内部仍可保留源码/预览切换能力，但它的归属不再是“代码编辑器附属能力”，而是右侧统一面板的一种内容类型。

### HTML 文件

点击 `html` 文件时，建议统一走 `resolveBrowserTarget(filePath)`：

1. 若文件可映射到开发服务地址，则打开对应 URL。
2. 若文件没有开发服务地址，但可通过本地静态代理地址访问，则打开代理 URL。
3. 若上述解析都失败，则回退到 `code` 视图查看源码。

默认策略是“优先浏览器预览，失败再回退源码编辑”。

### 聊天区与 Markdown 链接

当前聊天 Markdown 链接使用 `target="_blank"`。重构后应改为：

1. 识别 `http://`、`https://`、`localhost` 等可导航链接。
2. 阻止默认新开页行为。
3. 调用 `openUrlInRightPane(url, source)`。
4. 右侧自动切到 `browser` 视图。

## URL 解析与导航规则

### 地址栏输入

地址栏建议支持以下规则：

1. 输入完整 `http://` 或 `https://` 时直接访问。
2. 输入 `localhost:5173`、`127.0.0.1:3000` 时自动补齐为 `http://`。
3. 输入项目内静态预览地址时直接访问。
4. 输入明显不是 URL 的字符串时，不在第一版中扩展为搜索引擎查询，直接提示格式无效。

### 导航状态

浏览器视图至少应维护：

1. 当前 URL
2. 地址栏草稿值
3. 页面标题
4. 是否正在加载
5. 是否可后退
6. 是否可前进
7. 最近一次加载错误

## 错误处理

### 页面不可嵌入

当目标站点禁止被嵌入时，右侧浏览器应展示明确错误态，包含：

1. 失败说明
2. 当前目标 URL
3. 重试按钮
4. 可选的“复制链接”动作

### HTML 目标解析失败

当点击 `html` 文件但无法解析为可访问页面时：

1. 自动回退到 `code` 视图。
2. 可在右侧顶部显示一次性提示，说明当前打开的是源码而非浏览器预览。

### 本地页不可达

当本地开发地址连接失败时，提示：

1. 地址无法访问
2. 可能原因，如本地服务未启动
3. 刷新重试入口

## 测试策略

### 单元测试

1. `openFileInRightPane` 对不同扩展名的路由判断。
2. `resolveBrowserTarget(filePath)` 的解析与回退逻辑。
3. 地址栏 URL 规范化逻辑。
4. 浏览器导航状态更新逻辑。
5. 聊天链接点击改走右侧浏览器而非外开页面。

### 组件测试

1. 右侧面板根据不同 target 渲染正确视图。
2. 点击 `html` 文件时自动切到浏览器视图。
3. 点击普通文件时仍进入代码编辑。
4. 地址栏回车后触发导航。
5. 加载失败时展示错误态。

### 集成测试

1. 从文件树点击 `html` 文件到右侧浏览器展示的完整链路。
2. 从聊天消息点击链接到右侧浏览器展示的完整链路。
3. 浏览器视图下的后退、前进、刷新操作链路。

## 落地顺序

1. 把 `useEditorSidebar` 提升为通用右侧面板状态模型。
2. 引入 `RightPane` 容器和按类型路由的内容渲染层。
3. 先迁移现有代码编辑器打开链路，保证普通文件行为不退化。
4. 抽离 Markdown 为右侧面板的一种独立视图。
5. 新增浏览器视图和地址栏、导航控件。
6. 接入 `html` 文件解析与浏览器打开策略。
7. 改造聊天区链接点击逻辑，统一走右侧浏览器。
8. 补齐测试与错误态。

## 风险与取舍

1. 若宿主环境只能提供 `iframe`，部分外网站点可能无法内嵌，这是首版最大兼容性风险。
2. 若 `html` 文件与开发服务 URL 的映射规则不稳定，需接受部分文件先回退源码模式。
3. 右侧面板职责提升后，现有编辑器相关命名和组件边界需要同步整理，否则新旧语义会混杂。

## 结论

本设计建议把右侧区域正式升级为统一内容面板，并在其中新增可交互的小型内嵌浏览器。首版以单页、可导航、可交互、统一入口为目标，优先解决“右侧统一打开网页和 `html` 预览”的产品诉求。在此基础上，代码编辑和 Markdown 预览将一并纳入统一的右侧视图模型，后续扩展其他预览类型时无需再次重构整体架构。
