# Claude Agent V2 Conversation-Shell Run-Core 收口设计

## 背景

当前项目已经具备一套新的 Claude Agent V2 骨架，包括 `conversation / run / event` 模型、V2 仓储、事件投影和新的聊天壳 UI。但在实际运行中，旧聊天链路仍然保留着部分路由和状态决定权，导致新会话、继续会话和临时会话切换时仍可能出现串线、回跳和“没反应”的问题。

现阶段最重要的工作不是继续扩展能力，而是把主入口收口到同一条确定性链路上：

- 新会话只能走 V2
- 继续会话只能走 V2 的 conversation 绑定
- 旧链路只能读，不再写、不再决定路由

本设计的目标是把这些收口规则明确下来，作为后续实施和验证的唯一标准。

## 目标

1. 新会话创建只走 V2 的 `conversation + run` 流程。
2. 继续会话只根据真实 `sessionId` 创建新 `run`。
3. `new-session-*` 只作为短暂过渡态存在，不能再主导提交目标。
4. 旧聊天链路保留历史展示能力，但不再决定新消息发送路由。
5. 错误必须显式进入 run 生命周期，不能再静默吞掉。

## 非目标

- 不在本轮重构多 provider 抽象。
- 不删除全部旧链路代码，只做只读化收口。
- 不改造与 Claude 主链路无关的右侧面板、文件树或其它产品模块。
- 不为历史会话做复杂迁移，只要求新会话逻辑稳定。

## 方案选择

本轮收口采用方案 A：入口单轨化，旧链路降级为兼容层。

### 方案 A：入口单轨化，旧链路只读

新会话、继续会话、失败重试统一走 V2。旧的 `useChatSessionState`、`useChatRealtimeHandlers` 和 `sessionStore` 继续存在，但只承担历史展示和兼容读取，不再决定“把消息发给谁”。

优点：

- 能最快止住串线问题。
- 改动边界清晰。
- 新旧职责不会继续互相污染。

缺点：

- 需要明确切断旧链路的写权限。
- 中间态会同时存在旧展示和新投影，收口期间需要保持测试覆盖。

### 方案 B：双写双读

旧新链路同时工作，再靠后置校验选结果。

优点：

- 表面迁移风险低。

缺点：

- 会继续保留多个状态源争抢控制权。
- 不适合当前“会话跳回错误项目”的问题场景。

### 方案 C：分阶段 Feature Flag

通过开关灰度切换到 V2。

优点：

- 适合大范围发布控制。

缺点：

- 不能解决当前状态源混乱的根因。
- 会延长双轨并存时间。

结论：采用方案 A。

## 收口原则

### 1. 提交入口只有一个

前端 composer 的提交目标只能由 V2 负责决策。

优先级规则固定为：

1. `selectedSession`
2. `agentConversationId`
3. `session runtime binding`
4. `currentSessionId` 只作临时兜底

只要 V2 已经给出有效 `sessionId`，旧 session 逻辑就不能再改写提交目标。

### 2. 临时 session 只能是过渡态

`new-session-*` 只允许存在于首次提交后、真实 `sessionId` 还没回来的短窗口里。

一旦后端返回真实 `sessionId`：

- 前端必须立刻把视图绑定切到真实 session。
- 旧的临时 session 不再继续抢占当前显示态。
- 之后的消息只允许在同一个 conversation 下创建新 run。

### 3. 旧链路只读化

旧链路只保留以下能力：

- 历史消息读取
- 会话列表展示
- 旧会话回放

旧链路不再拥有以下能力：

- 决定新消息发送目标
- 创建新的运行时绑定
- 反向覆盖 V2 conversation 状态

### 4. 错误必须进入 run 生命周期

任何失败都必须体现在 run 事件和 run 状态里，不能只靠 transport 层吞掉。

对于旧坏会话，统一提示：

`该旧会话已无法继续，建议新建会话后重试。`

## 入口设计

### 新会话

当当前页面没有有效 V2 session 时，提交动作必须：

1. 创建或恢复当前 session 记录。
2. 创建首个 run。
3. 创建或恢复 Claude runtime session。
4. 通过统一事件协议回推 `run.started / assistant.message.delta / run.completed / run.failed` 等事件。

### 继续会话

当当前页面已经绑定到某个 session 时，提交动作必须：

1. 只创建新的 run。
2. 复用当前 session 的 runtime。
3. 不再重新生成一套独立会话上下文。

### 失败重试

对于无法继续的旧会话或错误会话，提交入口不能尝试偷偷回退到旧消息链路，而应：

- 显式展示失败
- 保留已完成的历史内容
- 提供“新建会话重试”路径

## 状态绑定设计

状态切换必须遵守单向规则：

`temporary session -> real session -> conversation-bound session`

禁止出现以下反向覆盖：

- 真实 session 被旧临时状态重新覆盖
- 当前项目被旧会话列表反向改写
- 已绑定 session 再退回到纯临时 session 视图

### 当前状态权威源

收口后只保留三类权威源：

- `sessionId` 决定当前 V2 对话上下文
- `runId` 决定当前执行事实
- `event store + projection` 决定界面显示

`currentSessionId` 只允许作为临时态存在，不再是权威源。

## 旧链路边界

### 允许保留

- 旧历史消息渲染
- 旧 session 回放
- 旧消息缓存
- 已有项目的只读查看能力

### 需要收回

- 新会话发送目标决定权
- 新 run 的创建路径
- session 与 conversation 的反向绑定
- 任何会把当前页面切回别的项目的逻辑

## 错误与降级

### 新会话失败

显示明确的 run failed 状态，并保留已生成内容。

### 继续会话失败

如果失败原因是旧坏会话不可继续，则显示明确文案，不再静默失败。

### 绑定丢失

当旧的 conversation runtime binding 丢失时，前端只读展示历史，不继续发送到旧 session。

## 测试与验收

### 必测场景

1. 在 `html` 项目新建会话并发送消息，确认不会跑到 `cloudcli`。
2. 新会话从 `new-session-*` 切换到真实 session 时，视图不回跳。
3. 已绑定 session 继续追问时，仍然复用同一条 session。
4. 旧坏会话失败时，明确出现“建议新建会话后重试”的提示。

### 验收标准

- 新会话提交路径只有一条。
- 旧链路不再影响新消息路由。
- 临时 session 不再抢权。
- 错误信息可见且可解释。

## 实施顺序建议

1. 先收口提交入口，确保新会话只走 V2。
2. 再收口状态绑定，保证 `new-session-*` 不会反向覆盖真实 session。
3. 然后把旧链路写权限收掉，只保留历史展示。
4. 最后补回归测试，确认 `html` 项目不会再串到 `cloudcli`。
