# 全项目滚动条统一设计

## 背景

当前 `cc-ui` 项目中的滚动区域主要有两类：

- 已封装的统一滚动容器 [`ScrollArea`](/Users/zhanglt21/Desktop/ccui0414/cc-ui/src/shared/view/ui/ScrollArea.tsx)
- 组件内部直接使用的 `overflow-y-auto`、`overflow-x-auto`、`overflow-auto`

这导致项目中的滚动条表现并不统一：

- 不同区域会直接暴露系统默认滚动条
- 聊天区、右侧预览区、列表区、弹窗区的滚动条粗细和视觉风格不一致
- 鼠标不在滚动区域时，滚动条仍然持续占据视觉注意力
- 后续新增页面如果继续直接写 `overflow-auto`，滚动体验会进一步分散

本次设计的目标，是在不引入真正虚拟列表渲染的前提下，统一项目中的“滚动条外观和交互规则”，让滚动区域默认更克制，在用户靠近时再出现更细、更轻、更精致的滚动条。

## 目标

- 全项目统一滚动条视觉风格
- 默认弱化滚动条存在感，鼠标进入容器或键盘聚焦时再显示
- 统一纵向和横向滚动条尺寸、圆角和配色
- 保持现有滚动行为，不改造成真正的虚拟列表或窗口化渲染
- 将 [`ScrollArea`](/Users/zhanglt21/Desktop/ccui0414/cc-ui/src/shared/view/ui/ScrollArea.tsx) 强化为标准滚动容器入口，便于后续渐进收口

## 非目标

- 不实现真正的虚拟滚动或大列表窗口化渲染
- 不在本轮强行替换项目中所有原生滚动容器结构
- 不重写聊天区、右侧预览区、设置页等现有布局层级
- 不引入新的第三方滚动条库

## 推荐方案

采用“全局样式统一覆盖 + `ScrollArea` 标准入口增强”的方案。

### 第一层：全局样式统一

在 [`src/index.css`](/Users/zhanglt21/Desktop/ccui0414/cc-ui/src/index.css) 中补充滚动条设计令牌和全局规则，使项目内大多数原生滚动容器先统一获得一致的滚动条外观和交互：

- 默认状态下，滚动条轨道和滑块接近隐藏
- 鼠标移入可滚动容器时，滚动条淡入显示
- 容器本身或容器内部元素获得键盘焦点时，同样显示滚动条
- 横向与纵向滚动条均使用统一的细窄尺寸

### 第二层：标准容器增强

增强 [`ScrollArea`](/Users/zhanglt21/Desktop/ccui0414/cc-ui/src/shared/view/ui/ScrollArea.tsx)，使其承担以下职责：

- 提供统一的滚动容器类名
- 为 hover / focus 可见性提供稳定选择器
- 保留现有触摸滚动体验
- 作为后续新页面和高频场景的默认滚动容器

第一轮不大面积替换项目内所有 `overflow-auto` 容器，仅优先统一视觉与交互，再在后续迭代中逐步将核心区域收口到 `ScrollArea`。

## 备选方案与取舍

### 方案 A：仅做全局 CSS 覆盖

优点：

- 改动最少
- 能快速让多数滚动区域统一风格

缺点：

- 缺少标准容器入口
- 后续新增页面仍可能继续分散实现
- 个别复杂区域难以做局部增强

结论：可以快速见效，但不利于后续长期维护。

### 方案 B：全局样式统一 + `ScrollArea` 增强

优点：

- 第一轮即可覆盖大多数滚动区域
- 同时建立标准滚动容器入口
- 风险低于“全量结构替换”
- 便于后续逐步收口

缺点：

- 第一轮在代码层面仍不是所有容器都完全统一到同一组件

结论：推荐采用。

### 方案 C：逐个区域替换成统一滚动组件

优点：

- 长期看结构最整齐
- 更容易针对局部复杂场景精修

缺点：

- 首轮改动量大
- 回归风险高
- 易引入高度塌陷、sticky 失效、内边距变化等布局问题

结论：不适合本轮一次性推进。

## 交互设计

### 基础规则

所有滚动区域遵循以下交互规则：

1. 普通状态下
   滚动条应尽量弱化，接近隐藏，但不影响鼠标滚轮、触控板和触摸滚动。

2. 鼠标移入容器内时
   滚动条显示，滑块透明度和对比度增强。

3. 容器或容器内部元素获得键盘焦点时
   滚动条显示，以保证键盘导航用户可以感知当前滚动区域。

4. 鼠标离开且容器失焦时
   滚动条回到默认弱化状态。

### 可访问性要求

- `focus-within` 必须与 `hover` 一样触发滚动条显示
- 不依赖 `hover` 作为唯一显示条件
- 不改变原本容器的键盘可滚动能力

## 视觉设计

推荐滚动条参数如下：

- 纵向滚动条宽度：`6px`
- 横向滚动条高度：`6px`
- 滑块圆角：最大圆角
- 默认轨道：透明
- 默认滑块：低透明度中性灰
- `hover/focus` 滑块：较默认态更深一档，但保持低存在感

视觉目标：

- 比系统默认滚动条更细
- 仅在需要时出现
- 不形成明显“整条灰槽”
- 浅色和深色主题下都能保持克制且可见

## 技术设计

### 全局样式令牌

在 [`src/index.css`](/Users/zhanglt21/Desktop/ccui0414/cc-ui/src/index.css) 的根变量中补充滚动条设计令牌，建议包括：

- `--scrollbar-size`
- `--scrollbar-radius`
- `--scrollbar-thumb`
- `--scrollbar-thumb-hover`
- `--scrollbar-thumb-active`
- `--scrollbar-track`

深色模式下提供对应覆盖值。

### 统一选择器策略

需要定义一组统一选择器，以兼容两种场景：

- 原生可滚动容器
- `ScrollArea` 内部的实际滚动视口

建议通过一类轻量 class 或 data attribute 标识“可应用统一滚动条规则的容器”，例如：

- `.ui-scrollbar`
- `[data-scroll-container]`

其中：

- `ScrollArea` 默认附带该标识
- 核心业务容器可逐步显式接入该标识
- 对于全局能稳定匹配的原生滚动区域，可通过通用规则覆盖

### `ScrollArea` 增强方向

[`ScrollArea`](/Users/zhanglt21/Desktop/ccui0414/cc-ui/src/shared/view/ui/ScrollArea.tsx) 建议调整为“双层职责明确”的结构：

- 外层负责圆角裁切、交互状态、可见性控制
- 内层负责实际滚动、触摸滚动、统一滚动条样式挂载

增强点包括：

- 默认加上统一滚动条 class / data attribute
- 外层或内层支持 `group` / `focus-within` 触发显示
- 保留 `WebkitOverflowScrolling: touch`
- 为后续特殊滚动场景预留扩展类名入口

## 首轮落地范围

### 优先通过全局样式覆盖的区域

- 聊天消息区
- 右侧预览和表单区
- 设置页主内容区
- Git 面板的列表区和历史区
- 任务面板和任务详情区
- 各类弹窗中的长内容区
- 横向可滚动的标签条、tab 条、代码块区域

### 优先收口到 `ScrollArea` 的区域

- 左侧项目 / 会话列表
- 文件树主滚动区
- 后续新增的长列表或标准侧栏区域

### 本轮明确不做的重构

- 不逐文件替换所有 `overflow-auto` 为 `ScrollArea`
- 不调整复杂容器的 DOM 结构来追求绝对统一
- 不引入独立的 JS 滚动条控制器

## 风险与处理

### 风险 1：滚动条出现时引发布局抖动

处理策略：

- 优先采用透明轨道与窄尺寸策略，减轻宽度变化带来的感知
- 尽量避免通过额外 padding 模拟滚动条占位
- 若个别区域仍抖动，局部补充专用样式修正

### 风险 2：部分第三方或特殊区域不完全受全局规则控制

处理策略：

- 对核心容器显式挂载统一 class / data attribute
- 必要时在组件局部补充滚动条样式

### 风险 3：不同浏览器滚动条能力不一致

处理策略：

- Chromium / WebKit 路径使用精细样式
- Firefox 使用 `scrollbar-width` 和 `scrollbar-color` 作为兼容降级
- 验收标准以“细、浅、hover/focus 可见”为主，不强求完全一致动画效果

## 验收标准

- 默认状态下，滚动条视觉存在感显著降低
- 鼠标移入滚动区域后，滚动条显示清晰但不厚重
- 滚动容器获得 `focus-within` 时，滚动条也会显示
- 横向和纵向滚动条风格一致
- 浅色和深色主题下都保持可见且克制
- 不引入明显布局抖动
- 不改变现有滚动逻辑和内容可达性
