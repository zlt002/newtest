# Visual HTML Editor 接入设计

## 背景

当前 `cc-ui` 已经具备完整的 HTML 文件打开链路：

- 文件树可识别 `.html/.htm` 文件
- 右侧面板通过 `RightPaneTarget` 路由到源码编辑或浏览器预览
- `BrowserPane` 支持同源 iframe 预览
- HTML 预览地址可通过 `/api/projects/:projectName/preview/...` 生成

本次目标不是引入新的页面存储格式，也不是替换现有编辑器体系，而是在现有 HTML/预览/编辑流程中增加一个可视化编辑视图。可视化编辑基于 GrapesJS，并直接回写原始 `.html` 文件。

## 目标

- 为现有 HTML 文件提供可视化编辑能力
- 保存时直接覆盖当前 `.html` 文件，不引入 JSON 中间格式
- 保留现有源码编辑器和浏览器预览能力
- 将 GrapesJS 作为右侧面板中的一种新视图，而不是独立子系统
- 在不适合可视化编辑的文件上平滑回退到源码模式

## 非目标

- 不替换 `CodeEditor`
- 不替换 `BrowserPane`
- 不引入 GrapesJS 远程存储或项目状态存储
- 不支持多页面 GrapesJS 项目模型
- 不对复杂模板文件提供完整兼容

## 推荐方案

采用“右侧新增 `visual-html` 视图”的方式接入 GrapesJS。

HTML 文件仍以原始文件文本为唯一真实来源。源码模式、可视化模式、预览模式只是在同一文件上的不同视图：

- `code`: 继续使用现有 `CodeEditor`
- `visual-html`: 使用 GrapesJS 编辑当前 HTML
- `browser`: 继续使用现有 `BrowserPane` 做最终预览

该方案与现有 `RightPaneTarget -> RightPaneContentRouter` 架构最契合，改动边界清晰，且不会把 GrapesJS 的内部状态扩散到主界面其他部分。

## 备选方案与取舍

### 方案 A：新增 `visual-html` 右侧视图

优点：

- 复用现有右侧多标签与路由体系
- 不污染 `CodeEditor` 职责
- 保持“源码 / 可视化 / 预览”边界清晰
- 初始化失败时容易回退到源码视图

缺点：

- 需要额外处理同一文件多视图之间的刷新与脏状态同步

### 方案 B：在 `CodeEditor` 内增加可视化模式

优点：

- 用户感知上仍是一个编辑器

缺点：

- 会显著加重 `CodeEditor` 的职责
- 脏状态、保存逻辑、切换逻辑会变复杂
- 不符合当前项目的右侧“多内容类型路由”模式

### 方案 C：用 GrapesJS 替换 HTML 预览

优点：

- 入口最直接

缺点：

- 混淆“预览”和“编辑”
- 会削弱现有 `BrowserPane` 的 iframe 预览和元素选择价值
- 不利于失败回退

结论：采用方案 A。

## 架构设计

### 新增类型与路由

右侧面板增加新的目标类型：

- `visual-html`

新增目标结构建议：

- `type: 'visual-html'`
- `filePath`
- `fileName`
- `projectName`
- `initialContent` 或延迟由组件内部读取
- 可选的版本校验信息，如内容摘要或最后加载标识

对应改造点：

- `src/components/right-pane/types.ts`
- `src/components/right-pane/view/RightPaneContentRouter.tsx`
- `src/components/right-pane/utils/rightPaneRouting.*`
- 右侧标签文案生成逻辑

### 新增组件边界

新增 `VisualHtmlEditor` 组件，职责限定为：

- 读取或接收当前 HTML 文件内容
- 初始化 GrapesJS
- 接收保存、关闭、刷新事件
- 导出最新 HTML 并写回原文件
- 管理可视化编辑器内部脏状态
- 在不可编辑场景中回退到源码模式

`VisualHtmlEditor` 不负责：

- 维护项目级页面状态
- 替代浏览器预览
- 管理跨文件依赖

### 与现有模块的职责分工

- `CodeEditor`: 源码编辑唯一承载
- `VisualHtmlEditor`: HTML 可视化编辑视图
- `BrowserPane`: HTML 最终预览
- 共享文件服务：负责读写文件、冲突校验、保存后广播刷新

## 数据流与保存策略

### 单一真实来源

`.html` 文件文本是唯一真实来源。

GrapesJS 不保存自己的 JSON 工程状态，不启用其项目存储能力。每次进入 `visual-html` 时，以当前文件内容初始化编辑器；每次保存时，导出 HTML 并覆盖原文件。

### 进入可视化编辑

1. 用户打开 HTML 文件
2. 文件可先进入源码视图
3. 用户切换到 `visual-html`
4. 系统读取该文件当前文本
5. 将文本注入 GrapesJS 初始化

### 保存流程

1. 用户在 GrapesJS 中编辑
2. 用户点击保存
3. 系统导出最新 HTML
4. 在保存前校验该文件是否仍与进入编辑器时一致
5. 若一致，直接覆盖原文件
6. 保存成功后通知其他同文件视图刷新
7. 已打开的预览页执行 reload

### 不做实时双向同步

第一版不实现源码视图与 GrapesJS 的实时双向同步，只采用：

- 进入时加载一次
- 显式保存时覆盖
- 保存后统一刷新其他视图

原因：

- GrapesJS 导出的 HTML 可能重排格式
- 实时同步会增加抖动、冲突和状态复杂度
- 当前右侧多标签架构更适合显式同步

## 冲突处理

### 场景 1：源码视图存在未保存修改

当用户准备进入 `visual-html` 时，如果同文件在源码视图中有未保存修改，则阻止直接进入，并提示：

- 先保存源码再进入
- 放弃源码未保存修改并进入

不允许带着未保存源码状态直接初始化 GrapesJS。

### 场景 2：可视化编辑期间文件被外部修改

在 `visual-html` 保存前，基于内容摘要、时间戳或最近读取标识进行校验。

若文件已变化：

- 阻止覆盖
- 提示用户该文件已被外部修改
- 提供重新加载可视化编辑器的路径

### 场景 3：GrapesJS 初始化失败

自动回退到源码视图，并提示该文件暂不适合可视化编辑。

## 文件适配范围

第一版只对以下文件开放可视化入口：

- `.html`
- `.htm`
- 纯静态或弱模板页面
- 少量内联样式页面

默认不开放或直接回退的文件：

- 大量内联脚本文件
- 带明显服务端模板语法的文件，如 `{{ }}`、`{% %}`、`<% %>`
- 带复杂框架运行时标记的 HTML
- 非完整页面、仅片段型的复杂模板

### 适配判断建议

增加一个轻量级“是否适合可视化编辑”的判定函数，基于文件扩展名和文本特征做启发式拦截。第一版允许误杀一部分边缘文件，也不要放开明显高风险文件。

## GrapesJS 嵌入策略

### 能力范围

第一版仅保留以下 GrapesJS 能力：

- Canvas
- 基础 block 拖拽
- 基础样式编辑
- 元素选中
- 层级查看
- HTML 导出并保存

第一版不启用：

- 远程存储
- 多页面管理
- 复杂插件市场
- 数据源功能
- 邮件模板专用高级能力
- 深度命令系统扩展

### UI 集成原则

GrapesJS 只渲染在 `RightPane` 内容区域内，不接管整个应用壳层布局。它需要适配当前面板尺寸、标签切换、展开收起和关闭行为。

## 预览联动

`BrowserPane` 保持原职责不变。

保存 HTML 后：

- 如果该文件已有 `browser` 预览标签，则触发该标签 refresh/reload
- 如果当前未打开预览，不强制新开标签

这样可以保留现有同源 iframe 和元素选择能力。

## 错误处理

需要覆盖以下错误场景：

- GrapesJS 初始化失败
- HTML 导出失败
- 文件写入失败
- 文件冲突校验失败
- 文件不适合可视化编辑

处理原则：

- 不影响源码编辑器可用性
- 明确提示原因
- 始终允许退回源码模式

## 测试策略

第一版重点测试接入边界，不测试 GrapesJS 内部行为。

### 单元测试

- `RightPaneTarget` 新类型路由测试
- HTML 文件类型识别测试
- 可视化入口可见性判断测试
- 冲突判断逻辑测试
- 保存后刷新广播测试

### 组件测试

- `RightPaneContentRouter` 正确渲染 `visual-html`
- 初始化失败时回退源码模式
- 保存成功后刷新关联 pane

### 集成验证

- 打开 HTML 文件并切换到可视化模式
- 编辑后保存并确认原文件被覆盖
- 切回源码模式确认文本更新
- 打开预览确认页面刷新
- 制造外部修改确认冲突提示生效

## 实施顺序建议

1. 扩展 `RightPaneTarget` 与路由
2. 搭建 `VisualHtmlEditor` 空壳并能正常渲染
3. 接入 HTML 文件读取与 GrapesJS 初始化
4. 打通保存回写原文件
5. 加入冲突校验与失败回退
6. 加入与源码视图、预览视图的刷新联动
7. 加入适配范围判断与入口限制
8. 补充测试

## 风险与控制

### 风险 1：复杂 HTML 被 GrapesJS 重写

控制方式：

- 第一版限制支持范围
- 对复杂模板默认回退源码

### 风险 2：与源码编辑器产生状态冲突

控制方式：

- 不做实时双向同步
- 进入前检查源码未保存状态
- 保存前检查文件是否外部变化

### 风险 3：右侧面板职责膨胀

控制方式：

- 将 GrapesJS 封装在独立 `VisualHtmlEditor`
- 不把逻辑塞入现有 `CodeEditor`

## 最终结论

本方案通过新增 `visual-html` 右侧视图，将 GrapesJS 作为现有 HTML 编辑链路中的一种可视化模式接入 `cc-ui`。系统仍以原始 `.html` 文件文本作为唯一真实来源，源码编辑、可视化编辑、浏览器预览三者职责分离，通过显式保存和刷新实现协同。

该方案与现有项目结构兼容性高、风险可控、上线边界清晰，适合作为第一版接入方案。
