# SingleFile 面向 CCUI 的网页蒸馏保存设计

## 背景

当前 `ccui` 在编辑普通 SingleFile 产出的 HTML 时，会遇到几个持续性问题：

- 页面结构过深，存在大量运行时包裹层、临时弹层容器和缓存节点
- `data-ccui-visual-html-canvas-style="true"` 容易积累大量碎片 `#id{...}` 规则
- 页面虽然能保存下来，但不够“编辑友好”，在 `ccui` 中选中、删改、保存都容易受到运行时结构影响
- 资源体积偏大，图片等资源会放大最终文件体积

目标不是做通用网页归档，而是新增一条“蒸馏保存”能力：输出一个更干净、静态、单文件、适合 `ccui` 可视化编辑的 HTML。

## 目标

- 输出单个 HTML 文件，尽量自包含，方便直接喂给 `ccui`
- 优先保持网页主要视觉结果接近原页面
- 优先适配后台系统、表单、表格、菜单、筛选栏、弹层等页面
- 不强求保留原始页面交互逻辑，采用“静态可编辑优先”
- 图片允许替换为占位符，但应保留尺寸、位置、边框、圆角、背景关系
- 尽量输出更浅的 DOM、更少 runtime 节点、更少碎片样式

## 非目标

- 不追求完整保留原始 JavaScript 交互
- 不追求首版支持所有营销官网和复杂动画页面的完美蒸馏
- 不追求把所有框架页面自动重建成完美语义化 HTML
- 不在首版引入 JSON DSL 或中间渲染协议

## 设计原则

- 平衡模式：优先保住主要视觉，再尽量做结构清理和可编辑化
- 静态可编辑优先：保留当前视觉结果，不保运行时行为
- 分阶段降噪：先冻结，再蒸馏，避免在采集时过早做高风险重写
- 面向 `ccui`：以 `ccui` 可稳定打开、选中、编辑、保存为结果导向
- 降级优先可用：复杂页面允许退回保守模式，避免“要么完美、要么失败”

## 总体方案

采用“双阶段平衡方案”。

### 阶段 A：静态冻结

目标：把当前网页冻结成一个“可蒸馏输入物”，固定当前视觉状态。

职责：

- 固定当前菜单展开态、弹层显示态、表格列宽、筛选栏布局、滚动区视觉状态
- 去除对运行时脚本继续执行的依赖
- 收集图片替换为占位符所需的尺寸与样式信息
- 只保留当前视觉结果需要的节点，不默认保留全部 runtime 隐藏缓存层

故意不做：

- 不做大规模 DOM 重建
- 不做最终 CSS 合并
- 不做深度语义重写

### 阶段 B：结构蒸馏

目标：把冻结结果转成更适合 `ccui` 编辑的单文件 HTML。

职责：

- 删除 runtime 垃圾节点
- 清理无效 CSS 和 orphan 规则
- 折叠无视觉贡献的 DOM 包裹层
- 收敛碎片样式，减少随机 id 依赖
- 做 `ccui` 兼容性检查，必要时触发降级

## 模块设计

建议新增独立的蒸馏模块目录，例如 `src/core/distill/`，避免直接污染现有普通保存链路。

### 1. `CaptureFreeze`

输入：当前页面上下文与保存选项  
输出：冻结态页面快照

职责：

- 冻结当前可见态
- 记录当前弹层、菜单、表格、筛选区、表单、滚动区信息
- 记录图片占位所需样式信息

建议挂载位置：

- 主要靠近 `src/core/content/content.js`

### 2. `EditableSnapshotBuilder`

输入：冻结态页面快照  
输出：初版静态 HTML

职责：

- 去掉脚本执行依赖
- 保留当前视觉所需 DOM
- 初步剔除浏览器扩展和无关注入痕迹
- 将图片替换为占位节点

### 3. `RuntimePruner`

输入：初版静态 HTML  
输出：删除 runtime 垃圾后的 HTML

职责：

- 删除临时弹层容器、框架挂载层、插件注入节点、离屏缓存层
- 不依赖固定 id 前缀做唯一判断
- 主要依据节点是否参与当前视觉、是否离屏、是否为纯缓存、是否对编辑结果有贡献

### 4. `StructureSimplifier`

输入：pruned HTML  
输出：更浅层、更稳定的 DOM

职责：

- 删除纯包裹层
- 合并无视觉贡献嵌套
- 保留后台页关键结构边界

重点边界：

- 左侧菜单组
- 顶部筛选栏
- 表格头/体
- 表单行
- 弹层主体
- 工具栏区块

### 5. `StyleCondenser`

输入：简化后的 HTML + CSS  
输出：最终单文件 HTML 所需样式

职责：

- 删除 body 中已不存在元素对应的 managed `#id{...}` 规则
- 将自动生成的碎片 `#id{...}` 尽量 inline 或合并
- 删除明显 runtime 的非持久样式
- 保留真正影响视觉的样式

### 6. `CcuiCompatibilityPass`

输入：蒸馏结果  
输出：最终可交付 HTML

职责：

- 检查是否能稳定进入 `ccui`
- 避免模板语法、外链脚本、结构异常导致 `ccui` 设计模式退回
- 对超复杂页面触发降级策略

## 数据流

1. 用户触发“蒸馏保存”
2. `content script` 进入 `CaptureFreeze`
3. 生成冻结后的静态 HTML
4. 后处理模块依次执行：
   - `RuntimePruner`
   - `StructureSimplifier`
   - `StyleCondenser`
   - `CcuiCompatibilityPass`
5. 输出单文件 HTML
6. 可选输出蒸馏报告

## 降级策略

采用三级降级，而不是单点失败。

### L2：平衡蒸馏

默认路径，面向后台页，兼顾视觉与可编辑性。

### L1：保守蒸馏

当页面过于复杂时：

- 减少 DOM 折叠
- 减少结构重写
- 优先保证视觉正确和 `ccui` 可打开

### Frozen Snapshot：冻结快照降级

当蒸馏风险过高时：

- 只输出冻结版静态页面
- 少量清 runtime 垃圾
- 保持可编辑，不追求结构最优

## 第一版范围

### 必须支持

- 后台系统页面
- 左侧菜单
- 顶部筛选栏
- 表格
- 表单
- 当前显示态弹层
- 当前展开态菜单
- 图片占位替换
- 单文件 HTML 输出
- `ccui` 设计模式可进入

### 可以弱化

- 原始 JS 交互保留
- iframe 深层交互还原
- shadow DOM 语义重建
- 图表、视频、音频、canvas 的完整保真
- 通用官网页面的极致结构美化

### 暂不做

- 通用 JSON 中间 DSL
- 完整语义化重建
- 复杂动画保真
- 所有网页类型统一高质量蒸馏

## 第一版验收标准

- 输入典型后台页面，输出单个 HTML 文件
- 输出结果能进入 `ccui` 设计模式
- 页面主视觉与原页面接近
- 表格、筛选栏、菜单、当前弹层状态基本正确
- 相比原始 SingleFile 结果，文件明显更小
- runtime 节点和 orphan CSS 明显减少
- `data-ccui-visual-html-canvas-style` 膨胀明显减轻
- 删除页面元素后，相关 managed CSS 能继续被清理

## 推荐实现顺序

1. 新增“蒸馏保存”入口
2. 实现 `CaptureFreeze`
3. 输出基础冻结 HTML
4. 清理 runtime 垃圾节点
5. 清理 orphan CSS / 收敛碎片 `#id` 规则
6. 做基础 DOM 折叠
7. 接入 `ccui` 兼容性检查
8. 加蒸馏报告与降级路径

## 风险与注意事项

- 视觉与结构天然存在冲突，不能用“最短 HTML”作为首要目标
- 不能过度依赖固定 id 前缀判断 runtime 节点
- 后台系统页面大量使用表格、弹层、离屏缓存，误删风险高
- `ccui` 设计模式资格判断较保守，蒸馏结果需要避开误判触发条件
- 首版更适合新增保存模式，不建议直接替换普通 SingleFile 保存

## 推荐结论

首版采用：

- 双阶段平衡方案
- 静态可编辑优先
- 单文件 HTML 输出
- 质量目标定为 `L2 平衡蒸馏`
- 优先支持后台系统/表单/表格页

这是当前风险、收益、落地速度最平衡的路径。
