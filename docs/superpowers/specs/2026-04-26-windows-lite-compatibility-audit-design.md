# Windows Lite 兼容性审计设计

## 背景

当前项目仍计划沿用现有的 `Windows Lite` 轻量发布方式，而不是切换到 Electron、Tauri、安装器或单文件 `exe`。这条发布链路的目标是生成一个可解压运行的目录包，Windows 用户通过双击 `start.vbs` 或 `start.cmd` 启动本地 Node 服务，再由默认浏览器访问本地地址完成使用。

项目最近经历了较大规模重构。虽然团队判断核心运行形态没有改变，即仍然由 `server/index.js` 提供本地服务、前端构建产物仍为 `dist/`，但现有 `windows-lite` 打包脚本是否仍然完全匹配当前仓库状态并没有被重新确认。

因此，本次工作不直接修改发布脚本，也不预设旧方案已经失效，而是先做一次针对当前仓库的兼容性审计，回答一个单一问题：当前项目是否还能继续沿用现有 `Windows Lite` 轻量发布链路出包。

## 目标

本设计的目标是定义一套清晰、可执行、可复核的 `Windows Lite` 兼容性审计方法，用于确认当前项目是否仍可继续使用现有轻量发布方案。

审计完成后，需要给出一个明确结论，而不是泛泛的“看起来问题不大”：

1. 可以继续沿用当前打包方案
2. 可以继续沿用，但需要先补少量适配
3. 当前重构已经让现有打包方案不再成立

本次工作的主判断标准偏向第一类问题，即“是否还能继续出包”，而不是追求列出所有潜在优化点。

## 非目标

本设计不覆盖以下内容：

1. 将 `Windows Lite` 改造为 Electron、Tauri 或其他桌面封装方案
2. 设计新的安装器、升级器或单文件 `exe` 交付方式
3. 对与 `Windows Lite` 无关的项目结构做大范围重构
4. 替代 Windows 真机验收
5. 围绕现有打包链路做功能扩展，只要不影响“能否继续出包”的判断，就不纳入本次范围

## 推荐方案

采用“分层兼容性审计”的方式，在不改动现有发布模型的前提下，依次确认架构假设、脚本假设和实际发布结果是否仍然成立。

推荐原因：

1. 最贴近本次真实目标，即确认当前方案是否还能继续用
2. 能区分“方案本身过时”与“局部脚本未同步”这两类问题
3. 比只做静态代码对照更可靠，因为它包含实际脚本执行结果
4. 比直接进入 Windows 真机测试更高效，适合作为当前阶段的首轮判断

## 方案对比

### 方案 A：只做静态对照

只阅读代码与脚本，人工判断 `windows-lite` 发布链路是否仍匹配当前仓库。

优点：

1. 成本最低
2. 可以快速发现显性路径错误、文件名变化或入口不一致

缺点：

1. 无法证明脚本真实还能跑通
2. 很容易漏掉依赖白名单、裁剪规则或安装阶段的运行时问题

### 方案 B：分层兼容性审计

这是推荐方案。

在静态对照基础上，进一步执行现有测试与发布脚本，对照产物结构给出结论。

优点：

1. 能同时覆盖“架构是否兼容”和“脚本是否还可运行”
2. 可以直接输出明确结论，适合当前决策目标
3. 改动最小，不会提前引入新的发布方案复杂度

缺点：

1. 仍然不能完全替代 Windows 真机验证
2. 如果审计发现问题，还需要后续单独制定修复计划

### 方案 C：直接做 Windows 真机启动验收

不作为本轮首选方案。

优点：

1. 离最终用户体验最近

缺点：

1. 成本最高
2. 如果前置脚本本身已经失配，真机验证会把问题混在一起，反而不利于定位
3. 不适合作为当前“先确认还能不能继续出包”的第一步

## 审计架构

本次审计分为四层，按顺序执行，任何一层出现阻断都需要记录并进入最终结论。

### 第一层：架构假设审计

目的：确认当前项目的核心运行模型是否仍满足 `Windows Lite` 轻量发布的基本前提。

检查重点：

1. 服务入口是否仍为 `server/index.js`
2. 前端构建输出是否仍为 `dist/`
3. 发布后运行方式是否仍是“本地 Node 服务 + 浏览器访问本地端口”
4. `windows-lite/start.cmd` 依赖的路径、日志目录、端口探测逻辑是否仍成立
5. `windows-lite/README.zh-CN.md` 是否仍与当前运行方式一致

### 第二层：发布脚本审计

目的：确认现有发布脚本对当前仓库结构和运行时依赖的假设是否仍然有效。

检查重点：

1. `scripts/build-release.mjs` 复制的目录与文件是否仍覆盖最小运行集合
2. `scripts/release-manifest.mjs` 中的运行时依赖白名单是否仍完整
3. `scripts/prune-release-node-modules.mjs` 与 `scripts/windows-lite-optimization.mjs` 的裁剪目标是否误删当前必需模块
4. `scripts/verify-release.mjs` 的必需路径、禁止路径和依赖校验逻辑是否仍与当前产物一致
5. `package.json` 中现有 `windows-lite` 脚本是否仍代表当前推荐发布路径

### 第三层：实跑验证审计

目的：验证现有链路不是“理论兼容”，而是确实仍能生成可用发布产物。

检查重点：

1. `windows-lite` 相关脚本测试是否通过
2. 发布脚本是否能从构建、安装、裁剪、校验一路执行完成
3. `release/windows-lite/` 产物结构是否完整
4. 发布结果是否与当前 `Windows Lite` 使用说明一致

### 第四层：结论分级

目的：把审计结果压缩成可直接决策的结论，而不是一组零散发现。

输出等级：

1. `绿灯`：当前项目可以继续沿用现有 `Windows Lite` 轻量发布方式
2. `黄灯`：总体方案仍成立，但要先补少量适配后再继续出包
3. `红灯`：当前重构已经让现有 `Windows Lite` 轻量发布模型失效，需要重新设计发布方式或大幅改造链路

## 验收标准

### 第一层通过条件

以下条件全部满足，架构假设审计才算通过：

1. `server/index.js` 仍能作为发布后的服务主入口
2. `vite build` 仍产出 `dist/index.html`
3. `windows-lite/start.cmd` 中引用的 `server/`、`dist/`、`logs/` 和本地端口探测路径仍有效
4. 发布后的用户运行方式仍然是启动本地服务并用浏览器访问本地地址

典型失败信号：

1. 入口文件已改名或迁移，但启动脚本未同步
2. 前端输出目录已变化
3. 健康探测地址或端口假设已失效
4. 启动脚本依赖的文件路径已被重构破坏

### 第二层通过条件

以下条件全部满足，发布脚本审计才算通过：

1. `build-release` 复制的目录仍覆盖运行所需最小集合
2. 运行时依赖白名单包含全部当前实际运行依赖
3. 裁剪规则不会误删当前运行所需模块
4. 校验脚本中的必需路径、禁止路径和依赖列表仍准确反映当前产物

典型失败信号：

1. 重构后新增了运行时依赖，但未进入白名单
2. 某些包在重构后从开发期依赖变成运行时依赖，但仍被裁剪
3. 校验脚本与现有产物结构不一致，导致假失败或漏校验

### 第三层通过条件

以下条件全部满足，实跑验证审计才算通过：

1. `windows-lite` 相关测试通过
2. 现有发布脚本执行完成，并输出成功校验结果
3. `release/windows-lite/` 中存在启动脚本、说明文档、`dist`、`server`、`shared`、`node_modules`
4. 产物目录中没有明显缺件，也没有保留不应交付的开发期内容

典型失败信号：

1. 构建通过但发布失败
2. 发布成功但校验失败
3. 安装、裁剪或校验阶段出现与当前架构不匹配的问题
4. 产物结构表面通过，但与当前运行方式存在明显断层

## 审计步骤

建议按以下顺序执行：

1. 阅读 `server/index.js`、`vite.config.js`、`windows-lite/` 启动脚本和 README，确认核心运行假设仍成立
2. 阅读 `scripts/build-release.mjs`、`scripts/release-manifest.mjs`、`scripts/prune-release-node-modules.mjs`、`scripts/verify-release.mjs`，确认脚本假设未过期
3. 运行 `windows-lite` 相关测试，确认已有保护网仍有效
4. 执行当前推荐发布脚本，优先以 `x64` 定向链路为主进行验证
5. 检查 `release/windows-lite/` 目录结构与文档说明是否一致
6. 按 `绿灯 / 黄灯 / 红灯` 输出结论，并在需要时给出最少量的后续修复建议

## 错误处理与结论原则

审计过程中如果出现失败，不应立刻得出“方案过时”的结论，而要先判断失败属于哪一类：

1. 架构性失配
2. 脚本未同步
3. 依赖白名单遗漏
4. 裁剪规则误删
5. 仅限当前环境或依赖安装异常

只有当失败已经触及发布模型本身，即“本地 Node 服务 + 浏览器访问”的基本前提不再成立时，才应判定为 `红灯`。

如果核心模型仍成立，只是脚本、白名单或校验细节需要补齐，则应判定为 `黄灯`，而不是误判为整套方案失效。

## 测试策略

本轮审计优先依赖以下验证手段：

1. 现有 `windows-lite` 相关单元测试和脚本测试
2. 当前发布链路的端到端执行结果
3. 发布产物目录结构检查

本轮不把 Windows 真机双击验证作为前置条件，但如果审计结论为 `绿灯` 或 `黄灯`，后续可以把真机验证作为上线前验收步骤。

## 交付物

本设计对应的交付物应包括：

1. 一次完整的 `Windows Lite` 兼容性审计结果
2. 明确的 `绿灯 / 黄灯 / 红灯` 结论
3. 如结论不是 `绿灯`，补充最小必要修复项列表

## 实施边界

本设计的直接后续工作应是编写实施计划，并按计划执行审计与验证。

后续实现阶段只应围绕“确认是否还能继续出包”这一目标展开，不应扩散为新的发布体系重构项目。
