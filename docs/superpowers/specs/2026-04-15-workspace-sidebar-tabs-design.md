# 左侧工作台三态切换设计

**日期**: 2026-04-15  
**作者**: Codex  
**状态**: 已确认

## 概述

当前左侧区域只承载项目与会话列表，而文件树与源代码管理位于其他区域，信息架构分散。目标是将左侧区域升级为统一的“工作台侧边栏”，在 `logo` 右侧提供 3 个一级 tab：

1. 项目
2. 文件
3. 版本

点击不同 tab 后，左侧主内容区切换为对应面板：

1. `项目`：现有项目与会话列表
2. `文件`：项目文件树
3. `版本`：项目源代码管理面板

桌面端与移动端采用同一套三态结构，仅在间距、字号和抽屉表现上做响应式适配。

## 目标

1. 在左侧 `logo` 右侧新增一级 tab 导航，支持 `项目 / 文件 / 版本` 三态切换。
2. 左侧主内容区按 tab 切换为项目列表、文件树、Git 面板。
3. 顶部工具栏按 tab 切换为对应工具栏，而不是固定显示项目工具栏。
4. 当前未选中项目时，`文件` 与 `版本` 提供清晰空状态提示。
5. 桌面端与移动端共享同一套交互模型。
6. 第一版尽量复用现有 `Sidebar`、`FileTree`、`GitPanel`，避免重写已有能力。

## 非目标

1. 第一版不调整右侧聊天区、编辑区的整体布局结构。
2. 第一版不重写 `FileTree` 与 `GitPanel` 的核心业务逻辑。
3. 第一版不引入新的全局状态库。
4. 第一版不对文件树与 Git 面板做大规模视觉重构，只做嵌入左侧所需适配。
5. 第一版不新增第四种工作台视图。

## 用户流程

1. 用户进入页面后，默认看到 `项目` tab。
2. 用户点击 `文件` tab。
3. 若当前已选中项目，左侧切换为该项目的文件树。
4. 若当前未选中项目，左侧显示空状态提示“请先选择项目”。
5. 用户点击 `版本` tab。
6. 若当前已选中项目，左侧切换为该项目的 Git 面板。
7. 用户切回 `项目` tab，原有搜索词、展开状态和会话列表保持不丢失。

## 方案对比

### 方案 1：在 Sidebar 内引入一级工作台视图

优点：

1. 复用现有侧边栏折叠、移动端抽屉、底部操作等基础能力。
2. `SidebarContent`、`FileTree`、`GitPanel` 都能以独立 panel 方式接入。
3. 状态边界清晰，后续扩展第四个 tab 成本低。

缺点：

1. 需要为左侧区域新增一层统一状态管理。
2. 需要为文件树与 Git 面板补充“嵌入左栏”时的 header 适配。

### 方案 2：把文件树与 Git 面板硬塞进现有 SidebarContent

优点：

1. 改动入口看起来少。

缺点：

1. `SidebarContent` 会演变成巨型条件分支组件。
2. 项目搜索、文件树、Git 面板状态容易互相耦合。
3. 后续维护与测试成本高。

### 方案 3：在 AppContent 层直接切三套左栏

优点：

1. 逻辑直观，各面板物理隔离。

缺点：

1. 折叠、移动端、底部操作和统一样式需要重复实现。
2. 公共行为分散，长期维护成本高。

### 采用方案

采用方案 1：在 `Sidebar` 体系内引入一级 `workspaceView` 状态，将左侧升级为统一的三态工作台容器。

## 信息架构

左侧区域统一为三层结构：

1. 一级导航层：`logo + workspace tabs`
2. 二级工具栏层：按当前 tab 切换
3. 主内容层：按当前 tab 渲染对应 panel

### 一级导航

位置：

- 放在 `logo` 右侧

内容：

1. 项目图标 tab
2. 文件图标 tab
3. 版本图标 tab

规则：

1. 当前激活 tab 需要有明显高亮态。
2. 移动端保留同一套 tab，不改成功能入口菜单。
3. icon 必须带可访问名称，文案可根据空间情况选择显示或仅作为 `aria-label`。

### 二级工具栏

按 `workspaceView` 切换：

1. `projects`
   - 项目模式下拉
   - 搜索框
   - 新建项目
2. `files`
   - 文件树标题或当前项目名
   - 文件树搜索与基础操作
3. `git`
   - 版本面板标题或当前项目名
   - 与左栏嵌入兼容的 Git 工具栏

原则：

1. 每个 tab 都有自己的 header/tool bar。
2. 项目专属控件不应出现在 `文件` 与 `版本` tab。
3. 第一版允许 `files` 与 `git` 工具栏较轻量，复杂交互继续放在各自面板内部。

### 主内容层

1. `projects`：现有 `SidebarProjectList` 和相关搜索结果视图
2. `files`：`FileTree`
3. `git`：`GitPanel`

## 状态设计

### 新增一级状态

在 [Sidebar.tsx](/Users/zhanglt21/Desktop/ccui0414/cc-ui/src/components/sidebar/view/Sidebar.tsx) 持有：

```ts
type WorkspaceView = 'projects' | 'files' | 'git';
```

建议默认值：

```ts
'projects'
```

用途：

1. 控制一级 tab 高亮
2. 决定二级工具栏渲染
3. 决定主内容层挂载哪个 panel

### 局部状态保留原则

各 panel 的状态继续本地管理，不合并成一个大对象：

1. `projects`
   - 搜索词
   - 搜索模式
   - 项目展开状态
   - 会话搜索结果
2. `files`
   - 文件树展开目录
   - 文件树搜索词
   - 视图模式
3. `git`
   - `changes/history/branches`
   - 文本换行设置
   - 展开 diff 状态

目标是确保 tab 切换不清空各自上下文。

## 组件设计

### 1. Sidebar

职责调整为：

1. 持有 `workspaceView`
2. 将当前选中项目透传给 `FileTree` 与 `GitPanel`
3. 统一处理折叠、移动端显示、底部公共操作

### 2. SidebarWorkspaceTabs

建议新增组件：

`src/components/sidebar/view/subcomponents/SidebarWorkspaceTabs.tsx`

职责：

1. 渲染 `项目 / 文件 / 版本` 三个一级 tab
2. 处理 icon、选中态、点击切换
3. 提供桌面端与移动端共用的语义层

### 3. SidebarHeader

从“项目专用 header”升级为“工作台 header”：

1. 第一行：`Logo + SidebarWorkspaceTabs`
2. 第二行：根据 `workspaceView` 渲染对应 toolbar

建议拆分二级工具栏组件：

1. `SidebarProjectsToolbar`
2. `SidebarFilesToolbar`
3. `SidebarGitToolbar`

### 4. SidebarContent

从“项目内容区”升级为“内容分发器”：

1. `projects` -> 现有项目列表逻辑
2. `files` -> `FileTree`
3. `git` -> `GitPanel`

### 5. FileTree

`FileTree` 已经可以独立渲染，但目前自带完整 header。第一版建议采用渐进接入：

1. 先直接嵌入现有 `FileTree`
2. 若出现双 header 或信息重复，再增加 `embedded` / `sidebar` 模式参数做裁剪

### 6. GitPanel

`GitPanel` 已经是完整独立面板，也自带 header 与二级 tabs。第一版建议：

1. 先直接嵌入现有 `GitPanel`
2. 左栏统一 header 只提供工作台级信息
3. Git 细粒度切换继续由 `GitPanel` 内部掌控

## 交互规则

1. 默认进入 `项目` tab。
2. 点击 `文件` / `版本` tab 时，不自动切换项目，只读取当前已选中的项目。
3. 若当前没有选中项目，则显示空状态：
   - “请先在项目页选择一个项目”
4. 从 `项目` tab 选中项目后，切到 `文件` / `版本`，使用该项目作为上下文。
5. 切换 tab 不清除原 tab 的局部状态。
6. 桌面端与移动端都使用同一套 tab 切换逻辑。

## 移动端设计

移动端也采用三态工作台，但要遵守以下约束：

1. 一级 tab 必须保留，不能退回到额外弹窗或更多菜单。
2. tab 高度与点击区域需满足触控尺寸要求。
3. 二级工具栏在空间不足时优先收缩文案与间距，不牺牲一级导航可用性。
4. `FileTree` 与 `GitPanel` 需要在移动端继续使用各自已存在的 `isMobile` 行为。

## 空状态设计

`files` 与 `git` 在没有选中项目时，显示统一空状态组件或统一文案风格。

建议文案：

1. 标题：`请先选择项目`
2. 描述：`先切换到项目页并选择一个项目，然后再查看文件或版本信息。`
3. 可选操作：`返回项目`

## 分阶段实施

### 阶段 1：建立左侧三态壳子

1. 新增 `workspaceView`
2. 实现 `SidebarWorkspaceTabs`
3. 在 `SidebarHeader` 渲染一级 tab
4. 保持 `projects` 为默认和唯一完整可用视图

### 阶段 2：接入文件面板

1. 在 `SidebarContent` 中挂接 `FileTree`
2. 处理无项目空状态
3. 验证文件打开仍走现有打开文件流程

### 阶段 3：接入版本面板

1. 在 `SidebarContent` 中挂接 `GitPanel`
2. 处理无项目空状态
3. 验证变更列表、历史、分支操作不受左栏嵌入影响

### 阶段 4：统一 header 与样式

1. 抽离三种 toolbar
2. 梳理桌面端与移动端样式一致性
3. 根据需要为 `FileTree` / `GitPanel` 增加 `sidebar` 模式适配

## 测试策略

### 单元测试

1. `workspaceView` 默认值与切换逻辑
2. 一级 tab 配置与激活态计算
3. 无项目时 `files` / `git` 的空状态分发

### 组件集成测试

1. 切换到 `files` 时渲染 `FileTree`
2. 切换到 `git` 时渲染 `GitPanel`
3. 从 `projects` 选择项目后再切换 `files` / `git`，上下文正确传递
4. 切换 tab 后各自局部状态保留

### 手工验证

1. 桌面端 tab 切换、折叠侧边栏、恢复侧边栏
2. 移动端抽屉打开、关闭、tab 切换
3. 文件树打开文件后右侧编辑器行为不回归
4. Git 面板执行刷新、切分支、查看历史不回归

## 风险与缓解

### 风险 1：FileTree 或 GitPanel 假设自己处于其他容器

缓解：

1. 第一版优先原样嵌入
2. 若发现布局冲突，再引入 `sidebar` 或 `embedded` 模式参数

### 风险 2：左栏宽度不足导致 Git 面板信息过密

缓解：

1. 第一版接受轻微信息拥挤
2. 后续再对 Git header 和列表项做左栏专属压缩布局

### 风险 3：移动端顶部高度增加

缓解：

1. 一级 tab 固定保留
2. 二级工具栏优先做紧凑排版
3. 必要时允许二级工具栏横向滚动，但不拆掉一级 tab

## 决策摘要

1. 左侧区域升级为统一三态工作台。
2. 一级 tab 放在 `logo` 右侧，桌面端与移动端都启用。
3. `文件` 与 `版本` 复用现有 `FileTree` 与 `GitPanel`。
4. 先做左栏壳子与 panel 切换，再做 header 和样式精修。
5. 第一版不改右侧主内容区结构。
