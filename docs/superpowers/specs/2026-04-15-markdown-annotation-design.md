# Markdown 可视化标注设计

**日期**: 2026-04-15  
**作者**: Codex  
**状态**: 已确认

## 概述

在 Markdown 可视化预览模式下，允许用户直接选中文本并通过右键菜单添加标注。标注数据单独保存为 JSON，不修改原始 Markdown 文件内容。系统需要支持一个 Markdown 文件保存多个标注区块，并在重新打开文件时准确回显高亮和标注内容。

本方案采用“完整版”实现路径：在 Markdown 渲染阶段保留源码位置信息，将渲染节点与 Markdown 源码位置绑定，再基于浏览器选区回溯到源码的行列范围。

## 目标

1. 用户可在 Markdown 可视化预览中选中内容并添加标注。
2. 标注保存为独立 JSON 文件，不污染原 Markdown 内容。
3. 一个 Markdown 文件支持多个标注区块。
4. 标注保存内容至少包含源码起止行列和标注文本。
5. 重新打开同一 Markdown 文件时，已有标注可正确回显。
6. 当文档内容变化导致标注无法可靠匹配时，系统应将其标记为失效，而不是错误贴到其他内容上。

## 非目标

1. 第一版不将标注同步到文件树、全局搜索、独立标注中心或项目级聚合视图。
2. 第一版不将标注写回 Markdown 文件正文。
3. 第一版不自动重定位失效标注。
4. 第一版不支持代码块内部、数学公式内部、原始 HTML 复杂嵌套内容的精细标注。

## 用户流程

1. 用户在右侧 Code Editor 中打开 Markdown 文件，并切换到可视化预览。
2. 用户在预览区选中一段文本。
3. 用户右键打开上下文菜单。
4. 当选区可稳定映射到 Markdown 源码时，菜单显示“添加标注”。
5. 用户填写标注内容并保存。
6. 系统将选区映射为 Markdown 源码行列范围，写入对应 JSON 文件。
7. 当前选中的文本片段立刻以高亮形式展示，并可查看、编辑、删除标注。
8. 下次打开该 Markdown 文件时，系统读取 JSON 并在预览区回显标注。

## 推荐方案

### 方案对比

1. DOM 选区直接反推源码：实现快，但复杂节点下容易漂移。
2. 渲染前保留 AST 源码位置信息并下传到 DOM：实现更复杂，但映射精度更高。
3. 只在源码视图做标注：精度高，但不符合“在可视化预览中选中并标注”的目标。

### 采用方案

采用方案 2。

核心原则：

1. Markdown 预览阶段必须保留源码位置信息。
2. 标注以源码范围为准，而不是以渲染后 DOM 位置为准。
3. 保存前必须做源码切片与选中文本的二次校验。
4. 无法稳定映射的选区不得保存为标注。

## 数据存储

### 存储位置

每个 Markdown 文件对应一个独立的标注 JSON 文件，推荐存储在项目根目录下：

`<projectRoot>/.ccui/annotations/<relativePath>.annotations.json`

示例：

- Markdown 文件：`docs/部署教程.md`
- 标注文件：`.ccui/annotations/docs/部署教程.md.annotations.json`

### 数据结构

```ts
type MarkdownAnnotation = {
  id: string;
  startLine: number;
  startColumn: number;
  endLine: number;
  endColumn: number;
  selectedText: string;
  note: string;
  quoteHash: string;
  createdAt: string;
  updatedAt: string;
};

type MarkdownAnnotationFile = {
  version: 1;
  filePath: string;
  fileHash?: string;
  annotations: MarkdownAnnotation[];
};
```

### 字段用途

1. `startLine/startColumn/endLine/endColumn`：标注主定位信息。
2. `selectedText`：用于显示摘要和匹配校验。
3. `note`：用户输入的标注内容。
4. `quoteHash`：选中文本的轻量哈希，用于辅助判断标注是否漂移。
5. `fileHash`：可选，表示保存时整篇文档的哈希，用于后续失效判断。
6. `annotations`：同一文件内多个标注区块的集合。

## 渲染与源码映射设计

### 核心思路

Markdown 文本在进入预览渲染时，系统需要保留 mdast 节点的源码位置信息，并在渲染后的 DOM 节点上暴露这些元数据。用户选中内容后，系统通过浏览器 `Selection` 和 `Range` 对象回溯到带有源码位置的数据节点，计算出最终的 Markdown 行列范围。

### 节点元数据

预览渲染后的可选中节点需要携带如下属性：

- `data-source-start-line`
- `data-source-start-column`
- `data-source-end-line`
- `data-source-end-column`
- `data-node-type`
- `data-node-id`

对于需要更细粒度映射的叶子文本片段，还需要保留局部文本偏移信息，例如：

- `data-text-start-offset`
- `data-text-end-offset`

### 选区回溯算法

算法分为两段：

1. 节点级定位
2. 节点内偏移定位

详细流程：

1. 获取浏览器当前选区 `window.getSelection()`。
2. 读取 `Range.startContainer/startOffset` 与 `Range.endContainer/endOffset`。
3. 从起点和终点容器分别向上查找最近的带源码位置数据的祖先元素。
4. 在命中的源码节点内，按渲染文本顺序累计字符偏移。
5. 将字符偏移换算为 Markdown 源码中的起止列偏移。
6. 结合节点自身的源码起止信息，计算最终的 `startLine/startColumn/endLine/endColumn`。
7. 根据计算出的源码范围，从原始 Markdown 文本中切片。
8. 将源码切片文本与用户实际选中的文本做归一化比较。
9. 当比较结果足够一致时，允许保存；否则提示用户当前选区无法稳定映射。

### 归一化校验

保存前的校验至少需要处理：

1. 连续空白折叠。
2. 不同换行风格统一。
3. 前后空格裁剪。
4. Markdown 渲染导致的部分可见文本与源码语法字符差异。

校验失败时不得保存，以避免错误标注。

## 支持范围

### 第一版支持

1. 标题
2. 段落
3. 列表项
4. 引用块
5. 表格单元格
6. 行内代码
7. 加粗、斜体、链接文本

### 第一版暂不支持

1. 代码块内部标注
2. 数学公式内部标注
3. 原始 HTML 复杂嵌套标注

## UI 与交互设计

### 右键菜单

仅在 Markdown 可视化预览区内启用标注相关上下文菜单。

行为规则：

1. 当存在有效选区且该选区可稳定映射到源码时，显示“添加标注”。
2. 当用户右键已标注区域时，显示“查看标注”“编辑标注”“删除标注”。
3. 当选区为空或无法稳定映射时，不显示添加标注动作。

### 标注输入

使用轻量弹层或气泡卡片收集标注内容，包含：

1. 选中文本摘要，只读。
2. 标注内容输入框，多行。
3. 保存按钮。
4. 取消按钮。

错误提示：

- “当前选区无法稳定映射到 Markdown 源码，请重新选择更完整的文本片段。”

### 标注展示

1. 已标注内容以浅色背景高亮显示。
2. 鼠标悬停可显示标注摘要。
3. 点击标注区可打开详情卡片。
4. 对同一片段的多个标注需支持叠加显示或数量提示。

### 标注管理

第一版仅在当前预览面板内管理标注，不增加独立侧栏。

详情卡片应支持：

1. 查看完整标注内容
2. 编辑标注
3. 删除标注
4. 查看源码范围信息

## 失效标注策略

当 Markdown 文档后续发生修改，导致已有标注无法可靠匹配到当前内容时，系统应采用保守策略：

1. 能精确命中的标注正常显示。
2. 无法精确命中的标注标记为失效。
3. 不自动重定位失效标注。
4. 在预览区域顶部或编辑器头部显示轻提示，告知存在失效标注数量。

推荐提示文案：

- “该文档有 N 条标注未能匹配到当前内容。”

## 实现拆分

### 阶段 1：预览渲染保留源码位置

目标：

1. 重构 `MarkdownPreview.tsx` 的渲染链路。
2. 保留 mdast `position` 信息。
3. 将源码位置信息注入可选中 DOM 节点。

### 阶段 2：选区映射引擎

目标：

1. 构建选区到源码范围的纯逻辑工具函数。
2. 实现节点级定位与节点内偏移计算。
3. 实现源码切片二次校验。

### 阶段 3：标注 JSON 读写层

目标：

1. 计算标注文件路径。
2. 读取和写入单文件标注 JSON。
3. 提供增删改查能力。
4. 支持目录自动创建和版本字段管理。

### 阶段 4：预览交互层

目标：

1. 监听选区与右键菜单。
2. 展示添加标注弹层。
3. 保存后即时高亮和展示详情。

### 阶段 5：标注回显与失效提示

目标：

1. 打开文件时读取对应标注 JSON。
2. 在预览中回显有效标注。
3. 对失效标注显示提示信息。

## 风险评估

1. 跨多个渲染节点的复杂选区映射可能出现偏差。
2. Markdown 渲染文本与源码文本并非严格一一对应，尤其在链接、表格、转义符号场景下更明显。
3. 文档修改后标注漂移是必然场景，若处理不谨慎容易误贴到错误内容。
4. 多标注高亮回显若采用全量 DOM 扫描，可能引发性能问题。

对应策略：

1. 通过源码切片校验兜底。
2. 第一版仅开放稳定映射的内容类型。
3. 失效标注不自动重定位。
4. 回显阶段先按节点范围筛选，再做局部匹配。

### 纯逻辑测试

覆盖：

1. Markdown 文件路径到 JSON 路径映射。
2. 标注 JSON 的增删改查。
3. 选区范围计算。
4. 失效标注判定。

### 渲染元数据测试

覆盖：

1. 标题、段落、列表、表格、行内代码等节点的源码位置信息透传。
2. 自定义组件渲染后元数据是否保留。

### 交互集成测试

覆盖：

1. 选中后可添加标注。
2. 保存后立即高亮。
3. 重新打开文件后标注仍可回显。
4. 编辑与删除动作生效。

## 落地顺序

1. 先完成本设计规格。
2. 基于规格输出实现计划。
3. 实现“渲染位置信息 + JSON store + 纯逻辑测试”。
4. 实现预览交互层。
5. 实现标注回显与失效提示。

## 验证标准

实现完成后，需要满足以下结果：

1. Markdown 预览中可选择文本并添加标注。
2. 标注数据以独立 JSON 形式落盘。
3. 一个 Markdown 文件可保存多个标注区块。
4. 重新打开文件后能正确回显有效标注。
5. 编辑文档导致无法匹配的旧标注不会错误高亮，而会被标记为失效。
