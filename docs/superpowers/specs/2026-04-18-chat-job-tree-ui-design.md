# Chat Job Tree UI Design

## 背景

当前聊天区对一次回答的执行过程采用了多套并存的表现方式：

- 输入区底部浮动 `处理中`
- 聊天流里单独出现 `Thinking...`
- `Task/子代理` 以局部卡片插入消息流
- 最终答案又回到普通 assistant markdown

这导致三个直接问题：

1. 过程态和完成态不是同一套界面，用户心智断裂
2. 子代理执行过程偏黑盒，用户只能等待，无法理解系统当前在做什么
3. 要理解执行链路时，用户常常需要借助 Network、日志或外部 tracing 工具

用户希望把整个过程组织成类似 tracing 工具的树状执行视图：结构化、有序、清晰，并且严格遵循“主代理包含子代理”的层级关系。

## 目标

把聊天区从“消息流 + 状态补丁”升级为“以主代理为根节点的一棵执行树”，统一展示：

- 主代理阶段
- 子代理派发与执行
- 工具步骤
- 告警与自动回退
- 汇总阶段
- 最终答案

## 非目标

本次设计不包含：

- 新增独立右侧 tracing 面板
- 改造底层 agent 协议或新增后端 tracing 存储
- 让所有原始工具日志默认完全展开
- 改造非聊天场景（如 Git、File Tree、Shell）的信息架构

## 核心设计结论

### 1. 一个回答 = 一个 Job Tree

一次用户请求在聊天区只对应一个根节点，根节点代表主代理的一次执行。整个回答从开始到结束都在这棵树中演进，而不是拆成多条视觉上互不相干的消息。

根节点固定为：

- `主代理`

其下依次挂载：

- `规划任务`
- `派发任务`
- `子代理节点`
- `汇总子结果`
- `最终答案`

### 2. 主代理是父节点，子代理必须嵌套

子代理不是平铺消息，不是与主代理同级的卡片，也不是树外独立块。

严格层级如下：

```text
主代理
├─ 规划任务
├─ 派发任务
│  ├─ 子代理 A
│  │  ├─ 当前步骤
│  │  ├─ 最近步骤
│  │  ├─ 告警 / 自动回退
│  │  └─ 结果摘要
│  └─ 子代理 B
│     ├─ 当前步骤
│     ├─ 最近步骤
│     ├─ 告警 / 自动回退
│     └─ 结果摘要
├─ 汇总子结果
└─ 最终答案
```

### 3. 过程态与完成态共用同一容器

完成后不切换成另一套 UI，不再把最终答案渲染成完全独立的普通 assistant 区块。

同一个 Job Tree 在生命周期内只是状态变化：

- `Queued`
- `Running`
- `Waiting`
- `Synthesizing`
- `Completed`
- `Failed`

### 4. 透明优先

默认展示：

- 主代理当前阶段
- 每个子代理的当前状态
- 每个子代理的当前步骤
- 每个子代理最近几步
- 告警与自动回退
- 子代理结果摘要
- 主代理汇总进度

默认折叠：

- 原始工具历史明细
- 完整原始工具结果
- 低价值重复事件

### 5. 完成后保留过程信息

完成后整棵过程树保留在结果上方或结果之前，不自动折叠成“查看过程”入口。用户可以直接回看：

- 主代理如何规划
- 派发了哪些子代理
- 每个子代理做了什么
- 哪些地方发生过回退或告警
- 主代理如何汇总

## 视觉与交互结构

## 根节点（主代理）

根节点负责提供一次回答的总体概览，包含：

- 用户请求摘要
- 当前总状态
- 已运行时长
- 子代理完成度
- 是否存在告警
- 是否允许停止

根节点默认展开。

### 根节点文案示例

- `主代理 · 派发两个子代理调研佛山的经济和天气`
- `运行中 · 2m15s · 2/2 子代理完成 · 正在汇总`

## 主代理阶段节点

主代理自己的步骤也进入树中，而不是只隐藏在顶部状态文案里。建议保留 3 个主阶段：

1. `规划任务`
2. `派发任务`
3. `汇总子结果`

这样用户可以明确理解：

- 系统先想清楚怎么拆任务
- 然后派发多个子代理
- 最后再回到主代理进行整合

## 子代理节点

每个子代理是 `派发任务` 节点下的直接子节点。

每个子代理节点默认显示：

- 子任务标题
- 子任务类型
- 执行时长
- token 使用量
- 当前状态
- 当前步骤
- 最近步骤
- 告警/自动回退
- 结果摘要

### 子代理默认展开层级

默认显示步骤，但原始工具历史折叠。

也就是说，用户一眼能看到：

- `WebSearch：定位 GDP 与产业结构`
- `WebFetch：校验失败，已自动降级`
- `Summary：输出经济摘要`

但不会默认看到所有原始 tool input / tool result 文本。

## 工具历史

原始工具历史采用双层入口：

1. 每个子代理卡内部可查看该子代理自己的完整工具历史
2. 整个 Job Tree 底部统一提供“完整执行日志”

默认只展示每个子代理的简版步骤，不直接展开完整工具明细。

## 异常与回退

### 可恢复异常

可恢复异常不终止整次 Job，而是作为树中的告警节点或告警块呈现。

例如：

- 某个来源校验失败
- 某次 WebFetch 超时
- 某次 WebSearch 结果质量不足后回退到备用来源

显示要求：

- 明确标记 `Warning` 或 `Fallback`
- 明确说明“系统已自动继续执行”
- 显示发生位置（哪个子代理 / 哪一步）

### 致命异常

只有当主代理或关键阶段无法继续时，Job Tree 进入 `Failed`，并在相应节点上显示：

- 失败发生在何处
- 是否可重试
- 是否已有部分结果可保留

## 权限等待

权限等待不只挂在输入框区域。进入等待态时：

- 根节点状态切换为 `Waiting`
- 对应节点显示“等待你的确认”
- Job Tree 中部插入待确认动作卡

这样用户不用盯着输入框，也能从执行树里理解“系统现在暂停在这里”。

## 最终答案

最终答案属于主代理树中的最后一个阶段节点，而不是树外的一条新消息。

显示形式：

- `最终答案`
  - markdown 正文

这样过程与结果天然连在一起，用户能理解“这段答案是如何从上面的执行树里长出来的”。

## 信息来源与数据映射

当前现有日志天然已经具有树状组织基础，应尽量复用，而不是重新发明一套抽象模型。

现有真实层级大致为：

1. user message
2. assistant thinking（主代理规划）
3. assistant tool_use: Task（主代理派发子代理）
4. user tool_result（子代理结果返回）
5. assistant thinking（主代理汇总）
6. assistant text（最终答案）

此外，现有消息体系中还已经存在：

- `task_started`
- `task_progress`
- `task_notification`
- `tool_use_summary`
- `subagent child tools`
- `warning / recoverable subagent errors`

本次前端设计应尽量基于这些已有结构做聚合与重组，而不是依赖新增协议。

## 推荐前端架构方向

聊天区需要从“线性消息列表”补强为“线性消息列表 + 树节点聚合渲染”。

推荐架构：

1. 保留消息流作为底层 source of truth
2. 在消息转换层把属于同一主代理执行链路的消息聚合成一个 `JobTreeViewModel`
3. 渲染层新增专门的 `JobTree` 容器组件
4. 现有 `Task/Subagent` 卡片逻辑下沉为 JobTree 的子节点渲染器

## 组件边界建议

建议新增或重组为以下视图单元：

- `JobTreeContainer`
- `JobTreeHeader`
- `JobTreeStageNode`
- `JobTreeSubagentNode`
- `JobTreeWarningNode`
- `JobTreeFinalAnswerNode`
- `ToolHistoryDrawer` 或 `ToolHistorySection`

## 体验原则

### 原则 1：用户不用看 DevTools

用户应直接在聊天区理解当前执行位置，不应依赖 Network、日志、外部 tracing 平台来理解系统状态。

### 原则 2：默认看懂，深入可钻取

默认显示必须足以回答：

- 现在谁在工作
- 做到了哪一步
- 卡在哪
- 是否在自动回退
- 还差什么结束

而不是只给一个“处理中”。

### 原则 3：树状优先于卡片感

UI 的第一职责是表达父子结构，而不是把每一层都做成互相割裂的漂亮卡片。

### 原则 4：最终答案是执行树的终点

最终答案不脱离执行树，否则用户仍会感觉“过程在一个地方，结果在另一个地方”。

## 风险与权衡

### 优点

- 组织逻辑与底层真实执行结构一致
- 子代理透明度显著提升
- 过程态和完成态统一
- 用户对失败、等待、回退的理解成本显著下降

### 风险

- 前端聚合逻辑会明显复杂化
- 现有线性 message rendering 需要调整边界
- 如果默认展开过多，会带来页面过长问题

### 风险缓解

- 默认展示步骤摘要，折叠原始工具历史
- 保持单根节点，避免多处重复显示状态
- 在转换层统一聚合，而不是在多个组件里各自推断

## 验收标准

以下条件同时满足，说明设计达标：

1. 一次回答在聊天区只出现一个主执行根节点
2. 子代理严格嵌套在主代理之下
3. 最终答案仍属于这棵树
4. 默认可看见每个子代理当前/最近步骤、告警、摘要
5. 完成后过程信息仍可直接查看
6. 用户无需打开 Network/日志即可理解系统当前状态

## 开放问题

当前设计已明确方向，但实现前仍需在计划阶段落细：

1. Job Tree 的聚合锚点应以哪类消息为准
2. 无子代理场景下，主代理树应如何最小化展示
3. 工具历史是嵌入树节点还是抽屉呈现
4. 超长执行树在移动端的折叠策略
